# 验证统计手册章节（自动版）

**目标**: 验证 `.reasonix/handbooks/stats-methods.md` 中某一章的 R 代码。

## 自动流程（三步）

```
① 自动查包版本 → ② 比对官方文档 → ③ 生成报告+建议复查日期
```

## 输入

任务参数中应包含要验证的**章节编号**（如 `2.2` 或 `2.5`）。

---

## 步骤 1：读取手册 + 提取包列表

用 `read_file` 读取 `.reasonix/handbooks/stats-methods.md`，定位到目标章节。

提取：
- R 代码块中的 `library(xxx)` 和 `xxx::function()` 调用
- 附录"包版本记录"中该章节涉及的版本号
- 当前标注的"最后验证"状态和"下次复查"日期

## 步骤 2：自动 CRAN 版本检查（核心）

**这一步自动执行，是验证的前提。**

对每个提取到的 R 包，搜索 CRAN 获取最新版本。

### 搜索方法

```
web_search("cran package glmmTMB version")
```

从搜索结果中提取：
- 最新稳定版版本号
- 发布日期
- 最近 3 次更新的内容摘要

### 版本比对逻辑

| 情况 | 标记 | 操作 |
|:-----|:-----|:-----|
| 手册版本 = CRAN 最新 | 匹配 | 无需操作 |
| 手册版本 < CRAN 最新（差 1 个小版本） | 有更新 | 检查 NEWS，看是否有破坏性变更 |
| 手册版本 < CRAN 最新（差大版本） | 需更新 | 必须更新代码，可能有 API 变更 |
| 手册中的函数已 deprecated | 需更新 | 找替代函数，重写代码 |

### 示例输出

```
| 包名 | 手册版本 | CRAN最新 | 发布日期 | 状态 |
|:-----|:---------|:---------|:---------|:-----|
| glmmTMB | 1.1.10 | 1.1.12 | 2026-03 | 有小更新 |
| vegan | 2.6-4 | 2.6-4 | 2025-08 | 匹配 |
| adegenet | 2.1.10 | 2.1.14 | 2026-05 | API 有变更 |
```

## 步骤 3：比对官方文档

对章节中的每个关键函数：
1. 搜索 CRAN 官方文档或 vignette
2. 确认函数签名、参数名、推荐用法是否匹配
3. 检查是否有 deprecated 的函数

## 步骤 4：检查权威来源

- **方法论文**：通过 `scholar_search_literature_graph` 搜索原始论文或最新综述
- **CRAN Task View**：如 "CRAN Task View: Mixed Models"
- **ima 知识库**：如果当前环境有 ima 工具，用 ima-smart-search 搜对应方法的代码作为比对样本


## 步骤 5：概率陈旧评分（替代固定复查周期）

不用固定周期。用概率模型计算某个包有多大概率已经过时：

P(stale) = w_freq * f(更新频率) + w_breaking * g(破坏性变更概率) + w_dep * h(用户依赖度)

| 因子 | 权重 | 计算方法 |
|:-----|:-----|:---------|
| 更新频率 f | 0.4 | 最近一年 CRAN 更新次数。月更 >=2: 陈旧快 |
| 破坏性变更概率 g | 0.35 | 读 NEWS 查是否有 API 变更。deprecated 标记: 紧急 |
| 用户依赖度 h | 0.25 | 是否在核心研究管线中使用。核心包: 影响大 |

映射到复查周期:
P > 0.7: 1 个月
0.4-0.7: 3 个月
0.1-0.4: 6 个月
< 0.1: 12 个月

差分验证: 只检查上次验证以来有更新的包和函数。不做全量检查。

## 步骤 6：生成验证报告

```
## 自动验证报告：章节 X.X

### 包版本状态
| 包名 | 手册版本 | CRAN最新 | 发布日期 | 状态 |
|:-----|:---------|:---------|:---------|:-----|
| xxx | x.y.z | a.b.c | YYYY-MM | 标记 |

### 代码检查
- 函数签名匹配：标记
- 文档推荐的替代写法：...
- 需更新的代码行：...

### 来源
- CRAN: [链接]
- Vignette: [链接]
- 原始论文: DOI

### 建议
- 下次复查：YYYY-MM-DD（基于包活跃度计算）
- 操作建议：无需操作 / 小修 / 章节重写
```

## 约束

1. 不要运行需要真实数据的代码
2. 如果 coderunner 不可用，仅做文档比对
3. 输出长度与验证结果成正比——没问题就短，问题多则详细展开

## 万物皆变

包在更新，函数在 deprecated，新方法在涌现。**今天的验证结果有效期为 3-24 个月**（取决于包活跃度）。建议在手册的"验证记录"表中更新日期和下次复查日期。
